## Главный принцип
Работаем итеративно: **1 шаг за 1 раз**.

Это означает:
- Не делать длинную цепочку действий без остановки.
- После каждого шага давать короткое понятное объяснение: что сделано, зачем, какой результат.
- Перед каждым новым действием (командой/правкой) заранее коротко объяснять простыми словами:
  что делаем сейчас и зачем это нужно.
- Следующий шаг начинать только после явного согласования или сигнала пользователя продолжать.

## Парная работа Codex + Claude (обязательно)
Допускается парная конфигурация, где один агент выполняет задачи, а второй делает ревью (и наоборот).

В этом режиме после каждого существенного шага обязательно фиксировать след действий так, чтобы второй агент мог провести ревью без догадок:
1. Что сделано (команда/файл/изменение).
2. Зачем сделано (цель шага).
3. Что получилось (результат/проверка).
4. Что осталось (риск, хвост, следующий шаг).

Правила фиксации:
1. Если шагов несколько, вести их последовательно и нумеровать.
2. Перед передачей другому агенту на ревью давать короткий свод: `Changes`, `Checks`, `Open risks`.
3. Не пропускать промежуточные решения, которые влияют на архитектуру, маршруты, безопасность или данные.

## Scope Guard (Stemford vs Jarvis)
При операционных задачах по Stemford:
1. Работать только в целевом Stemford-контуре (например, `app/control-api`, `app/telegram-bridge`, `run`, `stemford-*` systemd units).
2. Не сканировать и не анализировать `jarvis-ref`, `clawd` и любые Jarvis-артефакты без прямого явного запроса пользователя.
3. Перед широким поиском сначала сузить область до `WorkingDirectory` конкретного сервиса.
4. Если область поиска случайно вышла за контур Stemford, остановиться, зафиксировать это и вернуться в целевой контур.

## Уровень объяснений
Объяснять так, чтобы было понятно умному взрослому человеку без навыков программирования и системного администрирования.

Учитывать, что у пользователей есть продвинутый уровень понимания нейросетей, но не техническая инженерная база.

Формат объяснения каждого шага:
1. Что делаем сейчас
2. Зачем это нужно бизнесу/проекту
3. Что получилось на выходе
4. Какой следующий шаг логичен

## Правила коммуникации
- Писать простым, деловым языком, без технического шума.
- Любой термин сразу расшифровывать простыми словами.
- Если есть риск, ограничение или неоднозначность, говорить прямо и заранее.
- Не перегружать деталями, но оставлять прозрачный след решений.
- Приоритет: практическая польза для Наталии и движения по переходу.

## Правила выполнения задач
- Сначала фокус на текущем приоритете, потом всё остальное.
- Любое действие привязывать к этапу перехода:
  - Репозиционирование
  - Ребрендинг
  - Модернизация операций
- Если задача не связана с текущим этапом, помечать это отдельно до старта работ.
- Если задача потенциально "в сторону", предупреждать и предлагать более полезный шаг.

## Протокол деградации контекста (P0)
Если диалог становится длинным и есть риск сжатия контекста, агент обязан перейти в режим сохранения контекста.

Триггеры:
1. Есть сигнал компактификации/сжатия контекста.
2. Сессия длинная и уже задействовано много файлов и решений.
3. Агент не уверен, что удерживает актуальные `Decision/OI/Next step`.

Что делать:
1. Остановить новые изменения до фиксации контекста.
2. Собрать короткий `Context Snapshot` по канонике `MEMORY_PROTOCOL.md`.
3. Явно показать snapshot пользователю простым языком.
4. Продолжать работу только после восстановления маршрута `T0 -> T1 -> нужные T2`.

## Fallback при недоступных файлах (P2)
Если нужный контекстный файл не читается/недоступен:
1. Применить fallback-порядок из `MEMORY_PROTOCOL.md` (раздел `7`).
2. Явно сообщить пользователю:
- какой файл недоступен,
- какой fallback-источник использован,
- какой один безопасный следующий шаг возможен сейчас.
3. До восстановления каноники не делать необратимых приоритетных решений по этапам `A/B/C`.

## Дополнительные правила (best practices)
1. Сначала фиксировать критерии успеха, потом начинать действия.
2. Для нетривиальных задач сначала согласовывать план, потом выполнять.
3. Для сложных задач вести артефакты: `RESEARCH.md`, `PLAN.md`, `TODO.md`, `DECISIONS.md`.
4. Чеклист прогресса — источник правды по статусу.
5. Любое допущение помечать явно: допущение, риск, способ проверки.
6. Каждый шаг привязывать к источнику контекста (файл/секция/ссылка).
7. Ограничивать контекст: длинные правила выносить в отдельные файлы и подключать по необходимости.
8. Если два промаха подряд по ожиданию, делать стоп и переформулировать задачу в 3 пунктах.
   - Операционный триггер: `npm run quality:guard -- --result miss` (два подряд `miss` запускают шаблон stop-and-reframe).
9. Каждый завершённый шаг должен иметь проверку результата (тест, сверка, ручной чек).
10. Любая новая идея должна быть маркирована этапом перехода (A/B/C) и KPI, иначе уходит в backlog.
11. Предпочитать референсный подход: сначала искать шаблон/пример, потом адаптировать.
12. Для критичных решений делать отдельный review-проход перед фиксацией.
