## 2) Главный принцип

**Не автопилот, а управляемый штаб.**

Замы готовят и исполняют рутину. Наталья утверждает все решения с последствиями. Любое действие прозрачно: кто сделал, зачем, какой результат, какой риск. Любой шаг можно остановить командой «стоп».

## 5) Три режима взаимодействия Натальи с замами

### Режим 1 — «Принеси варианты»

Наталья задаёт тему. Замы готовят 2–3 варианта с плюсами и рисками. Наталья выбирает или предлагает свой.

> Пример: «Как объяснить родителям смену бренда?»
> Strategy + PMO приносят три тона (мягкий / прямой / через ценности).
> Наталья выбирает тон → PMO дорабатывает скрипт.

Когда использовать: новые инициативы, стратегические формулировки, неочевидные развилки.

### Режим 2 — «Проверь мою идею»

Наталья предлагает решение. Каждый зам даёт оценку со своей стороны.

> Пример: «Хочу запустить AI-курс в апреле»
> Strategy: «Это этап B, сейчас A — риск размыть фокус».
> Finance: «Запуск стоит X, при текущем оттоке не отобьётся за 3 мес.».
> PMO: «Нет готовых материалов, нужно 6 недель».
> Наталья решает: запустить / отложить / тестовый формат.

Когда использовать: у Натальи есть идея — нужна быстрая проверка на жизнеспособность.

### Режим 3 — «Делайте, я смотрю»

Замы работают автономно на задачах с понятным результатом. Наталья видит прогресс и может вмешаться в любой момент. Финальный артефакт — на её утверждение.

> Пример: заполнение конкурентной таблицы A7.
> Strategy работает, Наталья видит черновик, добавляет свои наблюдения, утверждает.

Когда использовать: аналитика, сбор данных, оформление — всё, где структура результата заранее понятна.

## 6) Уровни автономии (R0–R3)

| Класс | Тип действий | Автономия замов | Роль Натальи |
|-------|-------------|----------------|-------------|
| R0 | Чтение, аналитика, подготовка отчётов | Полная | Смотрит по запросу |
| R1 | Рутинные обновления задач (статус, блокер, next step) | Высокая | Видит в статус-срезе |
| R2 | Тактические изменения приоритетов, сроков, владельцев | Средняя | Подтверждает до исполнения |
| R3 | Стратегия, финансы, юр. действия, тексты для родителей | Только подготовка | Решение — только CEO |

**R2 и R3 не исполняются без явного CEO approved.**

Классификация R0–R3 зашивается в каждый SKILL.md как конкретная таблица «действие → уровень → что делать». Агент не угадывает уровень — он его знает.

### Что замы никогда не делают без слова Натальи

1. Не финализируют тексты для родителей.
2. Не принимают финансовых решений.
3. Не меняют приоритеты этапов A/B/C.
4. Не инициируют контакт с подрядчиками/юристами.
5. Не закрывают задачи без подтверждения результата.
6. Не пересматривают утверждённое CEO решение.

## 7) Совместная работа: где замы усиливают Наталью

Замы не заменяют экспертизу Натальи — они структурируют и ускоряют её.

| Ситуация | Как работает |
|---------|-------------|
| Наталья знает школу изнутри, зам — рынок снаружи | A7: она говорит «Logiscool слабее по практике» → strategy строит аргумент вокруг этого |
| Наталья чувствует родителей, зам структурирует | A6: она диктует, что важно сказать → PMO оформляет в FAQ и скрипты |
| Наталья видит риск, зам считает его | Finance: она говорит «боюсь потерять Пловдив» → finance даёт цифры: стоимость потери 10 учеников и варианты действий |
| Наталья перегружена, зам держит ритм | PMO напоминает: «A3 ждёт ваших 15 минут с понедельника» — без давления, просто видимость |

Ключ: **замы делают Наталью быстрее и увереннее, а не заменяют её суждение.**

## 8) Архитектура целевого контура

### Центральный принцип: Control API — хаб, AI-агенты — мозги

Control API (код, всегда онлайн) берёт на себя инфраструктуру: Telegram-мост, CEO-команды,
диспетчерскую, справки из базы. AI-агенты (включая orchestrator) — импульсные процессы,
просыпаются только когда нужно думать.

Код обрабатывает только то, что не требует ума. Всё остальное → orchestrator.

```
Telegram ←→ Control API (всегда онлайн, код, не AI)
                │
                ├─ Command Plane (код)
                │   CEO-команда? (/стоп, /пауза, /обстановка)
                │   → исполнить в коде, ответить мгновенно
                │
                ├─ Query Plane (код)
                │   Справка? (/статус A3, /что в работе)
                │   → SELECT из базы, ответить (ноль токенов)
                │
                ├─ Dispatch Plane (код)
                │   Зам закончил → кого будить дальше?
                │   → прочитать on_complete_wake_role из задачи/handoff_policies
                │   → разбудить следующего (данные, не if/else)
                │
                └─ Reasoning Plane (AI)
                    Всё остальное, без исключений
                    → разбудить orchestrator, передать сообщение как есть
                        │
                        ├→ Orchestrator (импульсный, как все остальные)
                        ├→ Strategy (импульсный)
                        ├→ Finance (импульсный)
                        └→ PMO (импульсный)
```

Orchestrator — координатор, но **не always-on процесс**. Его жизненный цикл такой же
импульсный (спит → разбужен → работает → засыпает), но у него особая роль:
Control API будит его для любого сообщения, не попавшего в Command/Query/Dispatch.
Strategy, Finance, PMO будятся только через orchestrator или dispatch.

**Целевая архитектура**: Control API (Telegram-бот, always-on, код) + orchestrator
(импульсный AI-координатор) + специализированные замы (импульсные AI-исполнители).

### Четыре plane Control API

| Plane | Что делает | Как распознаёт | Принцип |
|-------|-----------|----------------|---------|
| **Command** | CEO-команды: стоп, пауза, resume | Только слэш-команда (`/стоп`, `/пауза`) — mutation требует явного префикса | Строгое распознавание. Никакого угадывания |
| **Query** | Справки из базы: статус, список задач | Слэш-команда (`/статус A3`) или exact match (read-only, безопасно) | Только чтение. Ноль мутаций, ноль токенов |
| **Dispatch** | Диспетчерская: зам закончил → кого будить | Событие завершения задачи (внутренний trigger) | Маршрут из данных: `handoff_policies (source_role, event_type, condition_expr → target_role)` + shortcut `on_complete_wake_role`. Поддерживает ветвление по условиям |
| **Reasoning** | Всё, что не попало выше | Отсутствие совпадения в Command/Query | Forward to orchestrator целиком |

### Пять защит от хрупкости

1. **Строгое распознавание команд.** Mutation-команды (`/стоп`, `/пауза strategy`) — **только через слэш-префикс**, чтобы исключить ложное срабатывание в живом чате. Read-only запросы (`/статус`, `/обстановка`) допускают exact match без слэша. «Может стоит остановиться?» — это НЕ команда, это → Reasoning Plane.
2. **Малейшая неоднозначность → orchestrator.** Если сообщение не попало ни в Command, ни в Query — не пытаться угадать в коде. Сразу forward to orchestrator.
3. **Диспетчерская в данных, не в коде.** Маршруты хранятся в таблице `handoff_policies` с условными правилами: `(source_role, event_type, condition_expr) → target_role, priority`. Поле `on_complete_wake_role` в задаче — shortcut для простых случаев. Для ветвлений (например, «KPI-риск высокий → сразу CEO gate») — condition_expr в handoff_policies. Добавление нового маршрута = INSERT в таблицу, не правка if/else.
4. **Идемпотентность и дедупликация.** Два раздельных механизма:
   - **Dedup** (защита от дублей): таблица `processed_triggers (trigger_id UNIQUE, processed_at)`. Повторный webhook с тем же trigger_id → skip.
   - **Retry** (повтор после ошибки): отдельный code path `retryRun(failedRunId)`. Создаёт новый `agent_run` с `retry_of_run_id` → id упавшего run, `attempt_number` +1. Не проходит через dedup, т.к. это не новый trigger, а переиспытание существующего. Max 3 attempts (после — dead-letter → CEO).
5. **Всё в actions_log с correlation_id.** Любое действие любого plane — запись в лог. Один correlation_id на весь цикл от сообщения Натальи до ответа.

### Четыре слоя (реализация)

```
┌─────────────────────────────────────────────┐
│  OPERATOR LAYER                             │
│  Command Plane + Query Plane (код),         │
│  approval inbox, дайджесты (timer → AI)     │
├─────────────────────────────────────────────┤
│  CONTROL LAYER                              │
│  Approvals, hard critic, role boundaries,   │
│  R0–R3 gates, handoff policy matrix         │
├─────────────────────────────────────────────┤
│  RUNTIME LAYER                              │
│  Heartbeat-сервис, Dispatch Plane,          │
│  run state machine, atomic checkout,        │
│  session persistence, environment preflight │
├─────────────────────────────────────────────┤
│  BUSINESS LAYER                             │
│  A1–A9 задачи, goals, KPI,                 │
│  SOUL.md + SKILL.md per deputy              │
└─────────────────────────────────────────────┘
```

### Что обрабатывает код (Command + Query + Dispatch)

| Plane | Пример | Почему код, а не AI |
|-------|--------|---------------------|
| Command | `/стоп`, `/пауза strategy`, `/обстановка` | Кнопки. Мгновенное исполнение |
| Query | `/статус A3`, `/что в работе` | SELECT из базы. Ноль токенов |
| Dispatch | Зам закончил задачу → кого будить | Маршрут из данных, не суждение |
| Query | Пульт директора (по команде `/обстановка`) | Агрегация из DB, шаблон |

### Что уходит к orchestrator (Reasoning Plane)

| Тип | Пример | Почему AI |
|-----|--------|-----------|
| Бизнес-вопросы Натальи | «Как нам позиционироваться?» | Нужна классификация и маршрутизация |
| Синтез результатов | Собрать выводы 3 замов в decision pack | Нужно суждение, не формула |
| Сложная классификация | Непонятно, кому маршрутизировать | Код не справляется |
| Итог недели | Пятничный человеческий пересказ | Нужно осмысление, не список |
| Stage gate | Готов ли этап A к закрытию | Нужна экспертная оценка |

## 9) Heartbeat-сервис (ключевой новый компонент)

### Суть

Каждый зам (включая orchestrator) — отдельный OpenClaw-профиль, который не работает постоянно.
Heartbeat-сервис — это «секретарь в приёмной», модуль внутри Control API, который управляет
запусками всех четырёх замов.

### Цикл работы

```
Триггер (назначение / таймер / ручной вызов / сообщение Натальи)
  → Control API принимает триггер
  → Heartbeat-сервис проверяет: не занят ли зам? среда готова? не дубль ли?
  → Создаёт запись agent_run (status: queued → running)
  → Запускает OpenClaw-профиль с контекстом (task_id + wake_reason + correlation_id)
  → Агент работает, пишет результат в задачу через Control API
  → Агент завершается → heartbeat-сервис фиксирует результат (success / error / timeout)
  → Control API решает: нужен следующий зам? синтез? → диспетчерская логика
```

### Реализация

Node.js модуль в составе Control API (не отдельный процесс). ~200–300 строк.

### Что heartbeat-сервис делает (11 обязанностей)

1. **Принимает триггеры** — assignment, timer, on_demand, mention.
2. **Дедуплицирует** — два пути:
   - **Дубль webhook** (тот же trigger_id, нет флага retry) → проверяет `processed_triggers` → skip.
   - **Явный retry** (heartbeat сам решает после error/timeout, или CEO вызывает `/retry <run_id>`) → создаёт новую запись в `agent_runs` с `retry_of_run_id` = id упавшего run и инкрементным `attempt_number`. `processed_triggers` не блокирует, т.к. retry идёт через отдельный code path (`retryRun(failedRunId)`), а не через `acceptTrigger(triggerId)`.
3. **Проверяет параллелизм** — max 1 run на роль. Если занят → очередь или coalesce.
4. **Проверяет среду (preflight)** — OpenClaw CLI, SOUL.md, SKILL.md, DB, Telegram bridge. Не готово → run = error с причиной.
5. **Создаёт agent_run** — запись в DB со статусом, trigger, correlation_id.
6. **Запускает агента** — spawn OpenClaw-профиля с правильным system prompt и контекстом.
7. **Контролирует таймаут** — `max_run_timeout_sec` (default: 300). Превышение → SIGTERM → status = timeout.
8. **Фиксирует результат** — success / error / timeout + token_usage + error_message.
9. **Обеспечивает идемпотентность** — checkout, complete, approve безопасны при повторном вызове.
10. **Ведёт dead-letter счёт** — после 3 неудачных запусков по одной задаче → `needs_human_review` + уведомление CEO.
11. **Уведомляет Control API** — при завершении run → Control API решает, кого будить следующим (диспетчерская) или отправить результат Наталье. При ошибке/dead-letter → Telegram CEO напрямую.

### Guardrails

- `max_runs_per_day_per_role`: 20 (default). Защита от runaway.
- `max_run_timeout_sec`: 300 (default). Агент не может работать бесконечно.
- `max_consecutive_failures`: 3. После этого — dead-letter.

### Таблица agent_runs

```sql
CREATE TABLE agent_runs (
  run_id TEXT PRIMARY KEY,
  agent_role TEXT NOT NULL REFERENCES roles(role_id),
  task_id TEXT REFERENCES tasks(id),
  trigger TEXT NOT NULL,           -- 'assignment' | 'timer' | 'on_demand' | 'mention'
  trigger_id TEXT UNIQUE,          -- дедупликация
  correlation_id TEXT,             -- сквозной ID на весь цикл
  wake_reason TEXT,
  status TEXT NOT NULL,            -- 'queued' | 'running' | 'success' | 'error' | 'timeout' | 'cancelled'
  started_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  ended_at TIMESTAMPTZ,
  token_usage INTEGER,
  error_message TEXT
);
```

## 10) Inter-agent communication (через задачи)

### Принцип

Замы общаются **только через задачи в Control API**. Никаких side channels.

- Control API получает запрос Натальи → будит orchestrator для классификации → orchestrator создаёт задачу с `assignee=strategy`.
- Heartbeat-сервис будит strategy → strategy работает → завершает задачу с summary.
- Control API видит завершение → диспетчерская логика: нужен ещё кто-то? нужен синтез? → будит следующего зама или orchestrator для синтеза.
- Orchestrator собирает decision pack → Control API отправляет Наталье в Telegram.

### Atomic checkout

```
POST /tasks/:id/checkout  {"actor_role":"strategy"}  → 200 или 409 (уже занято)
POST /tasks/:id/release   {"actor_role":"strategy"}  → разблокировка
```

Один агент — одна задача. Гонки невозможны by design.

### Сквозной correlation_id

Один ID на весь цикл: trigger → agent_run → actions_log → telegram.
Любой инцидент расследуется за минуты: `SELECT * FROM actions_log WHERE correlation_id = 'X'`.

## 11) Параллелизм и тайминг

### Принцип: параллельно всё, что не зависит друг от друга

Max 1 run **per role**, но разные роли работают одновременно. Когда orchestrator
определил, что нужны несколько замов — Dispatch Plane будит их параллельно.

### Три паттерна исполнения

**Fan-out (параллельный)** — для независимых оценок:

```
Orchestrator (классификация, ~20 сек)
    │
    ├→ Strategy (анализ, ~40 сек) ──┐
    │                                ├→ Orchestrator (синтез, ~20 сек)
    └→ Finance  (KPI, ~30 сек) ─────┘

Итого: ~80 сек вместо ~110 сек последовательно
```

**Sequential (последовательный)** — когда второй зам зависит от первого:

```
Orchestrator → Strategy (варианты) → PMO (план реализации выбранного)
```

**Single (один зам)** — для простых вопросов:

```
Control API → Strategy (один ответ) → Наталье
```

Orchestrator решает, какой паттерн использовать. Dispatch Plane исполняет.

### Мгновенное подтверждение

Control API **сразу** отвечает Наталье, не дожидаясь результата:

> «Принято. Стратег и финансист работают, ожидай ~1–2 мин.»

Это <1 сек. Наталья не ждёт в пустоте. Decision pack приходит вторым сообщением.

### Ожидаемые тайминги

| Тип запроса | Цепочка | Время ответа |
|-------------|---------|-------------|
| CEO-команда (`/стоп`) | Код, без AI | <1 сек |
| Справка (`/статус A3`) | Код, без AI | <1 сек |
| Простой вопрос одному заму | Control API → один зам | ~30–40 сек |
| Вопрос с fan-out (2 зама параллельно) | orchestrator → 2 зама → синтез | ~1–1.5 мин |
| Полный decision pack (классификация + 2 зама + синтез) | orchestrator → fan-out → orchestrator | ~1.5–2 мин |

Команды и справки — быстрее, чем сейчас. Сложные решения — медленнее, но значительно глубже.

## 12) Handoff-протокол

### Стандартная цепочка

```
Наталья (запрос через Telegram)
  → Control API (мгновенное «принято»)
  → Orchestrator (классификация + маршрут)
    → Fan-out: профильные замы параллельно
  → Orchestrator (синтез в decision pack)
  → Control API → Наталья (решение)
```

### Правила

1. **Validate before call.** Каждый межагентный вызов — через `POST /handoff/validate`.
2. **Контракт передачи.** Каждый handoff содержит: вход, ожидаемый выход, допущения, correlation_id.
3. **Anti-loop.** Max глубина цепочки = 3, max retry = 2. Превышение → эскалация CEO.
4. **Прямые вызовы запрещены.** Finance↔strategy, pmo↔strategy — только через orchestrator.
5. **Каждое действие в логе.** Все handoff фиксируются в `actions_log` с correlation_id.

### Decision Pack (для R2/R3)

Единый шаблон. Каждый зам отдаёт в одном формате:

```
## Вопрос на решение
[Одно предложение]

## Варианты
A: [описание] — плюсы / минусы / влияние на учеников / влияние на финансы
B: [описание] — плюсы / минусы / влияние на учеников / влияние на финансы

## Рекомендация замов
[Какой вариант и почему — с атрибуцией: Стратег считает X, Финансы считают Y]

## Что будет, если не решить сейчас
[Конкретные последствия]

## Нужно от Натальи
[ ] Выбрать вариант / предложить свой / отложить
```

## 15) Контекст между сессиями

### Session persistence per task

```sql
CREATE TABLE agent_task_sessions (
  agent_role TEXT NOT NULL,
  task_id TEXT NOT NULL,
  session_data JSONB,
  updated_at TIMESTAMPTZ DEFAULT now(),
  PRIMARY KEY (agent_role, task_id)

```

Агент помнит, что делал по конкретной задаче. При следующем heartbeat — продолжает, не начинает с нуля.

### Memory cards

Кросс-задачный контекст (предпочтения Натальи, принятые решения). TTL 168 часов. Дополняет, не заменяет session persistence.

### Журнал решений

Каждое утверждённое CEO решение фиксируется: дата, причина, кто подготовил. Замы не пересматривают без нового запроса.

## 18) Production hardening (10 инвариантов)

### Этап 1 (вместе с runtime foundation)

1. **Дедупликация триггеров.** Два code path: `acceptTrigger(triggerId)` проверяет `processed_triggers (trigger_id UNIQUE)` — дубль webhook → skip. `retryRun(failedRunId)` — создаёт новый agent_run с `retry_of_run_id` + `attempt_number`, обходит dedup. Max 3 attempts → dead-letter.
2. **Лимит параллелизма.** Max 1 run на роль одновременно. Второй запрос → очередь или coalesce.
3. **Идемпотентность мутаций.** `checkout`, `complete`, `approve` безопасны при повторе.
4. **Сквозной correlation_id.** Один ID: trigger → agent_run → actions_log → telegram.
5. **Минимальные guardrails.** `max_runs_per_day_per_role` (20) + `max_run_timeout_sec` (300).

### Этап 2 (вместе с авто-handoff)

6. **Dead-letter очередь.** После 3 неудачных запусков → `needs_human_review` + уведомление CEO.
7. **Протокол CEO-команд.** Конкретные команды в Telegram с однозначным эффектом (см. раздел 12.3).
8. **Единый шаблон Decision Pack.** Все замы отдают в одном формате (см. раздел 11).

### Этап 3 (перед боевым запуском)

9. **Теневой запуск (3–5 дней).** Замы работают, но R2/R3 без авто-исполнения. Сверяем качество.
10. **Runtime smoke-тесты (S10+).** Отдельные smoke для heartbeat / checkout conflict / wakeup / dedup / dead-letter.

## 22) CI/CD контур: Claude Code + Codex на сервере

### Назначение

Автоматизация разработки и деплоя самой системы замов. Два AI-инструмента устанавливаются
на VPS и работают в связке — один пишет код, другой ревьюит и контролирует качество.

Подробная техническая реализация — в `PLAN_OpenClaw_Control_Plane.md` (§31 Three-Agent Gate).

### Суть

Три роли — **Executor** (Codex/GPT-5.4), **Reviewer** (Claude Opus), **Deployer** (bash-скрипт).
Четыре уровня fallback-ротации при исчерпании лимитов: Codex → Claude → Gemini → Qwen.

Цикл: `executor → commit → reviewer (P1=0?) → deployer`.

Все операционные детали — реестр инструментов, скрипты, ротация, установка, guardrails —
вынесены в **[`docs/VPS_CODERS_RUNBOOK.md`](../docs/VPS_CODERS_RUNBOOK.md)**.

### Связь с отделом замов

CI/CD контур обслуживает кодовую базу системы замов. Это разные слои:
- **Замы** (orchestrator, strategy, finance, pmo) — работают с бизнес-задачами Натальи через OpenClaw.
- **CI/CD** (Codex + Claude Code) — разрабатывают и деплоят код Control API, heartbeat-сервиса, дашборда.

Замы не знают о CI/CD. CI/CD не вмешивается в бизнес-логику замов.

## 24) Отложенные заимствования из AAF для контура кодеров (на потом)

Статус: Deferred (после стабилизации текущего CI/CD контура).
Приоритет: Medium.
Цель: повысить автономность и "человечность" взаимодействия без потери safety-gate.

### Что берем из AAF позже

1. Event-driven очередь задач с приоритетами (вместо линейного скриптового маршрута).
2. Явный lifecycle run-сессий (start/resume/interrupt/recover) с единым run_id.
3. Роль orchestrator как постоянного маршрутизатора событий, а не только shell-обертки.
4. Watchdog/self-healing для сервисов контура кодеров (автодиагностика + эскалация).
5. Стандартизованный handoff между executor/reviewer/deployer через task bus.
6. Разделение "мозга" и "исполнителя": reasoning в агенте, критичные действия через жесткие gate.
7. Единая модель памяти по задаче (короткая operational memory, без смешения с бизнес-памятью замов).
8. Ограниченная plugin-модель для кодерских задач (только whitelisted действия).

### Что НЕ берем из AAF в текущей реализации

1. Docker-in-Docker с пробросом docker.sock (слишком высокий риск для VPS).
2. Полный heavy stack памяти (GraphRAG + vector + SQL одновременно) для кодеров.
3. Полностью автономный self-modifying контур без обязательного human gate.
4. Userbot-режим Telegram (MTProto) вместо Bot API в прод-контуре.

### Условия старта этого блока

1. Текущий контур стабилен минимум 2 недели (без ложных SUCCESS и зависаний).
2. Все статусы цикла человекочитаемы и однозначны (SUCCESS/NO_DIFF/BLOCKED/... на русском).
3. Зафиксированы runbook и rollback для критичных шагов деплоя.
4. Покрыты preflight checks и алерты по ключевым отказам.

### Критерий готовности "сделано"

1. Оркестратор принимает естественный язык без хрупких шаблонов.
2. Внутри этапа агенты автономны, наружу обязательны только start/end сообщения владельцу.
3. Все handoff и решения трассируются в одном журнале событий.
4. Надежность не ниже текущего CI/CD baseline (smoke PASS, gate не ослаблен).
