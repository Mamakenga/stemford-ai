# План внедрения: Reliability-first + Kanban Control Plane

Статус: Draft for execution  
Версия: 1.0  
Дата: 2026-03-28  
Область: `app/control-api` + `telegram_bridge` + `dashboard` + runtime-пайплайн

## 1) Зачем этот план

Цель: превратить текущий набор сервисов в управляемую систему, где:
1. задачи не мешают друг другу;
2. процесс по задачам не «прыгает» через этапы;
3. «готово» ставится только после обязательных проверок;
4. ревью и риски фиксируются формально;
5. у каждого действия есть прозрачный след;
6. человек работает через удобный веб-интерфейс, а не только через команды.

Идея простыми словами:
1. Не просто «бот отвечает», а работает производственная линия.
2. У линии есть конвейер, журнал и пульт управления.

## 1.1) Focus lock: сначала coder factory

Чтобы не скатиться в универсальный task manager, фиксируем приоритет явно:
1. Этот план в первую очередь доводит до рабочего состояния `Coder Factory Dashboard`.
2. Под `Coder Factory` понимаем контур с ролями `executor / reviewer / deployer` и обязательными Start/End согласованиями владельца на этапах.
3. `Business Dashboard` для управляющего Stemford — это следующий слой, который делается уже после стабилизации coder factory.
4. Любая текущая работа по kanban/web UX должна отвечать вопросу: помогает ли это видеть и управлять pipeline `executor -> reviewer -> deployer`.

Практический смысл:
1. Сначала делаем систему, которая надежно делает код и деплой.
2. Потом этой системой делаем бизнес-контур "джарвисов".
3. И только потом обобщаем решение в reusable service.

## 2) Что уже есть как база

Уже реализовано в текущем контуре:
1. База статусов и задач в Control API.
2. Gate-механика Start/End approvals.
3. API для Kanban и health summary.
4. Веб-страница `/dashboard`.
5. Telegram-команды `/gate_start` и `/gate_end`.
6. Базовый actions/audit слой.

Это означает: мы не начинаем с нуля, а усиливаем и «доводим до промышленного состояния».

## 3) Шесть рабочих механизмов (ядро внедрения)

## M1. Изоляция задач (Task Isolation)

Что делаем:
1. Для каждого `task_id` закрепляем отдельный рабочий контекст выполнения:
- отдельный run-контекст;
- отдельные логи;
- отдельный набор служебных артефактов.
2. Запрещаем смешивание артефактов разных задач в одном run.

Зачем:
1. Одна «сломаная» задача не заражает остальные.
2. Проще разбирать инциденты.

Критерий готовности:
1. По `task_id` можно получить полный независимый trail: run -> logs -> checks -> решение.

## M2. Жесткая статус-машина (Strict State Machine)

Что делаем:
1. Фиксируем разрешенные переходы:
- `Backlog -> Ready -> In Progress -> Review -> Done`;
- отдельная ветка `Blocked` с явной причиной.
2. На API-уровне запрещаем нелегальные переходы.
3. Для плановых задач включаем порядок шагов:
- следующий шаг не стартует, пока предыдущий не закрыт.

Зачем:
1. Убираем хаос «перепрыгиваний».
2. Статус становится источником правды, а не чатом.

Критерий готовности:
1. Любая попытка нелегального перехода возвращает формальный reason code.

## M3. Quality Gate перед Done

Что делаем:
1. Вводим обязательный чек-лист перед `Done`:
- `review_p1_zero`;
- `required_checks_passed`;
- `smoke_passed` (если задача затрагивает runtime);
- `gate_end_approved` (если нужен end gate).
2. Сервер блокирует `Done`, если хотя бы один пункт не закрыт.

Зачем:
1. «Сделано» означает «проверено», а не «кажется работает».

Критерий готовности:
1. Нельзя перевести задачу в `Done` без формального PASS.

## M4. Review Loop P1/P2 как системное правило

Что делаем:
1. Вносим в карточку ревью список замечаний с уровнем:
- P1 (блокер),
- P2 (можно в follow-up).
2. Merge/Done блокируется при `P1 > 0`.
3. P2 автоматически уходит в backlog follow-up задач.

Зачем:
1. Критичное исправляется сразу.
2. Некритичное не тормозит поток.

Критерий готовности:
1. Для каждой задачи виден статус ревью и факт закрытия P1.

## M5. Полная трассировка (Audit Trail)

Что делаем:
1. Для каждого важного действия пишем event:
- кто;
- что;
- по какой задаче;
- когда;
- с каким результатом.
2. Стандартизируем типы событий:
- `task_state_changed`,
- `approval_requested/decided`,
- `gate_requested/decided`,
- `run_started/finished`,
- `chat_message_added`,
- `quality_gate_failed`.

Зачем:
1. Всегда понятно, что реально произошло.
2. Разбор сбоев перестает быть «угадайкой».

Критерий готовности:
1. Любую карточку можно восстановить по таймлайну событий.

## M6. Веб-пульт (Kanban + Chat + Gates)

Что делаем:
1. Развиваем текущий `/dashboard` до «операционного пульта»:
- колонки Kanban;
- карточка с таймлайном;
- встроенный чат задачи;
- кнопки `Request Start Gate`, `Request End Gate`, `Approve`, `Reject`.
2. Добавляем две роли просмотра:
- Owner (операционный),
- CEO (управленческий, без тех-шума).

Уточнение фокуса:
1. Первая целевая версия этого web-пульта = `Coder Dashboard`.
2. На нем должны быть прозрачно видны текущая роль, stage gate и результат этапа:
- `executor`
- `reviewer`
- `deployer`
3. Business-oriented kanban для управляющего — следующий контур, а не замена coder dashboard.

Зачем:
1. Управление становится человеческим, а не только командным.

Критерий готовности:
1. Основной цикл задачи проходит в вебе без обязательного ручного ввода Telegram-команд.

## 4) Пошаговый план внедрения

## Фаза A. Нормализация ядра процесса

Шаги:
1. Зафиксировать таблицу разрешенных переходов статусов.
2. Вынести в сервер единый валидатор переходов.
3. Добавить reason codes для отказов.

Артефакты:
1. Документ переходов статусов: `docs/TASK_STATE_MACHINE.md`.
2. Тесты API на разрешенные/запрещенные переходы.

Gate:
1. Нелегальные переходы отклоняются стабильно.

## Фаза B. Усиление gate-модели

Шаги:
1. Довести start/end gate до одинакового контракта.
2. Добавить явные причины reject.
3. Привязать `Done` к quality checklist.
4. Хранить обязательные и пройденные проверки прямо в задаче.

Артефакты:
1. Обновленные endpoint-контракты.
2. Smoke-сценарии «без approve не пройти».
3. Quality gate contract (`quality_checks_required` / `quality_checks_passed` / `quality_gate_failed`).

Gate:
1. Без закрытых условий `Done` невозможен.

## Фаза C. Review discipline P1/P2

Шаги:
1. Добавить модель review findings в задачу/run.
2. Ввести серверную блокировку на `P1 > 0`.
3. Реализовать авто-создание follow-up для P2.

Артефакты:
1. API ревью-замечаний.
2. Индикация P1/P2 в dashboard.

Gate:
1. P1 всегда = 0 перед завершением.

## Фаза D. Task isolation и run hygiene

Шаги:
1. Упорядочить хранение run-логов по `task_id/run_id`.
2. Добавить проверку «одна активная execution-сессия на task».
3. Укрепить retry/dead-letter с сохранением trail.

Артефакты:
1. Единая схема run-артефактов.
2. Операционные скрипты диагностики по `task_id`.

Gate:
1. Нет дублей параллельного выполнения одной задачи.

## Фаза E. Web UX: Kanban как основной пульт

Шаги:
1. Расширить карточку задачи: история, ревью, проверки, гейты, чат.
2. Добавить быстрые действия (approve/reject/gate request/reopen/block).
3. Добавить фильтры: assignee, stage, blocked, overdue.

Артефакты:
1. Обновленный `dashboard.html` + API адаптация.
2. UX smoke: 5 типовых сценариев владельца.

Gate:
1. 80% ежедневных операций выполняются через веб-пульт.

## Фаза F. Операционный контур надежности

Шаги:
1. Свести единый health view: API + bridges + runtime.
2. Добавить дежурные smoke-проверки после релиза.
3. Утвердить runbook инцидентов (bridge conflict, stuck task, failed gate).

Артефакты:
1. Health/diagnostic сценарии.
2. Обновленный OPS runbook.

Gate:
1. Типовые инциденты закрываются по инструкции без импровизации.

## 5) Матрица проверки качества (обязательная)

На каждом релизе проверяем:
1. State machine test pack.
2. Gate test pack (start/end + rejection reasons).
3. Review test pack (P1 block, P2 follow-up).
4. Kanban API contract.
5. Dashboard smoke (ручной пользовательский проход).
6. Log/audit completeness check.

PASS-критерий релиза:
1. Все обязательные тест-паки зеленые.
2. Нет `false success` в smoke.
3. В audit trail нет «пустых» критичных действий без actor/result.

## 7) Что считаем «полностью рабочим состоянием»

Система считается готовой, когда одновременно:
1. владелец ведет задачи через Kanban-пульт;
2. без Start/End gate важные шаги не проходят;
3. без `P1=0` задачи не закрываются;
4. любой результат имеет проверяемый audit trail;
5. при сбоях есть предсказуемое восстановление по runbook.
