# CONCEPT: Stemford AI Deputies Model (Codex)

Version: 1.0  
Date: 2026-03-17  
Owner: CEO (Наталья)  
Execution model: OpenClaw + Control API

## 1) Цель концепции

Собрать реалистичную модель работы AI-замов для Stemford, где:
1. Замы реально разгружают CEO.
2. CEO сохраняет управленческий контроль.
3. Совместная работа замов дает скорость и качество, а не хаос.

Ключевой принцип:
**Замы готовят и исполняют в рамках полномочий. CEO принимает ключевые решения.**

## 2) Бизнес-контур (3 эшелона)

Работа идет строго по этапам:
1. Эшелон 1: Репозиционирование (приоритет №1).
2. Эшелон 2: Ребрендинг.
3. Эшелон 3: Операционная модернизация.

Правило перехода:
следующий эшелон открывается только после закрытия gate предыдущего.

## 3) Роли замов и зоны ответственности

Состав замов:
1. `orchestrator` — маршрутизация, сбор итогов, контроль handoff-цепочек.
2. `strategy-gatekeeper` — стратегический фильтр, соответствие этапу, позиционирование.
3. `finance-kpi` — KPI-оценка, экономические эффекты, допущения и риски.
4. `pmo` — ритм исполнения, owner/deadline/status/blocker, weekly checkpoint.

Непересекаемые границы:
1. Strategy не утверждает финансы.
2. Finance не меняет стратегический приоритет этапов.
3. PMO не принимает стратегические или финансовые решения.
4. Orchestrator не подменяет решения CEO.

## 4) Режимы взаимодействия CEO с замами

### Режим 1: «Принеси варианты»
Замы приносят 2-3 варианта с плюсами/рисками. CEO выбирает.

### Режим 2: «Проверь мою идею»
CEO задает идею. Замы дают валидацию по своим осям (стратегия/финансы/исполнение). CEO решает.

### Режим 3: «Делайте, я смотрю»
Для рутинных задач с понятным результатом. Замы работают автономно в пределах правил, CEO видит статус и вмешивается по необходимости.

## 5) Уровни автономии (контрольный баланс)

| Класс | Тип решений | Автономия | Кто финализирует |
|---|---|---|---|
| R0 | Чтение, аналитика, отчеты | Высокая | Замы |
| R1 | Рутинные изменения задач и процессов внутри этапа | Высокая | Замы |
| R2 | Тактические изменения приоритетов/сроков пакетов | Средняя | CEO подтверждает |
| R3 | Стратегия, финансы, юр. риски, внешние коммуникации | Низкая | Только CEO |

Жесткое правило:
`R2/R3` не исполняются без явного `CEO approved`.

## 6) Что замы никогда не делают без слова CEO

1. Не финализируют тексты для родителей.
2. Не принимают решения с финансовыми последствиями.
3. Не меняют приоритеты этапов A/B/C.
4. Не инициируют внешние юридические действия.
5. Не закрывают стратегические задачи без подтверждения результата CEO.

## 7) Как замы взаимодействуют друг с другом

Стандартная цепочка:
`orchestrator -> профильный зам -> reviewer/critic -> orchestrator -> CEO (если R2/R3)`

Handoff-контракт обязателен:
1. Вход.
2. Ожидаемый выход.
3. Риск/допущения.
4. Срок.
5. Следующий владелец.

Anti-loop ограничения:
1. Ограничение глубины handoff.
2. Ограничение retry в цепочке.
3. Эскалация к CEO при превышении порогов.

## 8) Техническая реализация на OpenClaw + Control API

### Что уже является базой
1. Структурированный слой задач и статусов.
2. Approval gates и hard critic.
3. Tool-access policy по ролям.
4. Actions log, deploy/handoff trace.
5. Smoke-сценарии надежности.

### Что включаем как обязательный runtime-режим
1. Отдельные рабочие профили замов (`orchestrator`, `strategy-gatekeeper`, `finance-kpi`, `pmo`).
2. Единый вход CEO через Telegram.
3. Автоматизированные handoff-цепочки под капотом.
4. Decision pack для `R2/R3` вместо автопринятия решений.

## 9) Операционный ритм

Понедельник:
1. PMO отправляет недельный срез.
2. Фиксируются 2-4 решения, нужные от CEO.

В течение недели:
1. Замы работают автономно по `R0/R1`.
2. На `R2/R3` формируют decision pack и ждут решения CEO.
3. Блокеры эскалируются сразу.

Пятница:
1. Orchestrator отправляет короткий человеческий итог недели.
2. CEO подтверждает курс на следующую неделю.

По запросу:
1. CEO может в любой момент запросить черновик конкретного зама.
2. Команда `стоп/пауза/пересмотрите` мгновенно замораживает текущий шаг.

## 10) Stage A (репозиционирование): практическая раскладка

Пакет Stage A:
1. A1 Идентичность и миссия.
2. A2 УТП и позиционирование.
3. A3 ЦА и ценностное предложение родителям.
4. A4 Удержание учеников.
5. A5 Коммуникационный пакет для родителей.
6. A6 Конкурентный ландшафт.
7. A7 Ревизия программ.
8. A8 Юридический чеклист и карта рисков.
9. A9 Decision pack на CEO gate.

Распределение:
1. Strategy: A1, A2, A3, A6, A7.
2. PMO: A4, A5, CHK.
3. Finance: KPI-валидация и экономические риски по решениям.
4. Orchestrator: синтез и маршрут CEO-решений.

## 11) Принципы прозрачности и доверия

1. Любое важное действие оставляет след в логах.
2. Любое решение класса `R2/R3` имеет объяснение «почему».
3. Любое утвержденное CEO решение фиксируется с датой и причиной.
4. Замы не пересматривают утвержденное CEO решение без нового запроса.

## 12) Финальная формула управления

**Не автопилот, а управляемый штаб.**

1. Замы дают скорость, структуру и аналитику.
2. CEO дает направление, выбор и финальные решения.
3. Система обеспечивает контроль, совместную работу и предсказуемость.
